# Day 68 — Introduction to Ansible and Inventory Setup

## What is Configuration Management?
Configuration management means managing the configuration of servers after they are provisioned.
Once a server is ready, we need prerequisites and configurations installed to run our application
such as Docker, Nginx, Apache and other packages. Ansible is also used as a CD (Continuous Delivery)
tool in some organizations to deploy application code after infrastructure is provisioned.

Another key benefit is handling **drift** — if someone manually changes a config on a server,
CM tools detect and correct that drift, bringing the server back to the desired state automatically.

---

## Ansible vs Chef, Puppet, Salt

| Tool | Agent | Mechanism | Notes |
|------|-------|-----------|-------|
| Ansible | Agentless | Push | SSH based, faster TTM |
| Chef | Agent required | Pull | Agent pulls config from server |
| Puppet | Agent required | Pull | Agent pulls config from server |
| Salt | Agentless/Agent | Push + Pull | Can do both |

---

## Ansible Architecture

| Component | Description |
|-----------|-------------|
| Control Node | Machine where Ansible is installed and all commands are fired from |
| Managed Nodes | Target servers that get configured via SSH, no Ansible needed here |
| Inventory | File listing all managed nodes grouped by role |
| Modules | Individual units of work — install, copy, start, ping, etc. |
| Playbooks | YAML files that combine modules into full automation workflows |

**Why agentless?**
Ansible uses SSH push-based mechanism so it does not require any agent on managed nodes.
Ansible simply SSHes into the server and performs its work directly.

## Lab Setup

### Infrastructure
Provisioned using Terraform. Mixed AMI environment:

| Role | Hostname | AMI | SSH User |
|------|----------|-----|----------|
| Control Node | control-node-ubuntu | Ubuntu 22.04 | ubuntu |
| Web Server | worker-ubuntu | Ubuntu 22.04 | ubuntu |
| App Server | worker-centos | CentOS (AL2023) | ec2-user |
| DB Server | worker-amazon | Amazon Linux 2023 | ec2-user |

### Key Setup
Private key copied from local machine to control node:
~~~bash
scp -i ~/your-key.pem ~/your-key.pem ubuntu@<control-node-ip>:~/.ssh/
chmod 400 ~/.ssh/your-key.pem
~~~

### Ansible Installation (Control Node only)
~~~bash
sudo apt update && sudo apt install ansible -y
ansible --version
~~~

Ansible is installed only on the control node because it uses agentless push-based
SSH mechanism — no installation required on managed nodes.

## Inventory File — hosts.ini

~~~ini
# Ansible inventory - Auto-generated by Terraform
[all:vars]
ansible_ssh_private_key_file=~/keys/ansible-master-key
ansible_python_interpreter=/usr/bin/python3
ansible_host_key_checking=false

[servers]
control-node-ubuntu ansible_host=x.x.x.x ansible_user=ubuntu

[web]
worker-ubuntu ansible_host=x.x.x.x ansible_user=ubuntu

[app]
worker-centos ansible_host=x.x.x.x ansible_user=ec2-user

[db]
worker-amazon ansible_host=x.x.x.x ansible_user=ec2-user

[application:children]
web
app

[all_servers:children]
application
db
servers
~~~

**Note:** IPs redacted for public sharing.

**Key learning:** In a mixed AMI environment, define `ansible_user` per host
since Ubuntu uses `ubuntu` and Amazon Linux/CentOS uses `ec2-user`.

---

## ansible.cfg

~~~ini
[defaults]
inventory = hosts.ini
host_key_checking = false
remote_user = ec2-user
private_key_file = ~/keys/ansible-master-key
~~~

Once this file is created in the project directory, you no longer need
to pass `-i hosts.ini` in every command. Ansible reads config in this order:
1. Current directory `ansible.cfg`
2. `~/.ansible.cfg`
3. `/etc/ansible/ansible.cfg`

### 4. Install a package using dnf (AL2023)
~~~bash
ansible worker-amazon -i hosts.ini -m dnf -a "name=git state=present" --become
~~~
~~~
worker-amazon | CHANGED => {
    "changed": true,
    "results": [ "Installed: git-2.50.1-1.amzn2023.0.1.x86_64" ]

**Lesson learned:** AL2023 does not support the `yum` module (requires Python 2).
Use `dnf` module instead for Amazon Linux 2023.

~~~bash

~~~


## command vs shell Module

| Feature | command | shell |
|---------|---------|-------|
| Pipes `\|` | Not supported | Supported |
| Redirects `>` | Not supported | Supported |
| Environment variables `$HOME` | Not supported | Supported |
| Shell scripts | Not supported | Supported |
| Safety | More secure | Less secure |

**Rule of thumb:** Use `command` by default. Switch to `shell` only when
you need pipes, redirects, or environment variables.

~~~bash
# This will FAIL with command module
ansible all -m command -a "cat /tmp/hello.txt | grep Ansible"

# This will WORK with shell module
ansible all -m shell -a "cat /tmp/hello.txt | grep Ansible"
~~~

## --become Flag

`--become` is equivalent to `sudo` in Linux. It escalates privileges to root
on the managed node. Required for tasks like:
- Installing packages
- Managing services
- Writing to system directories

~~~bash
# Without --become — will fail with permission denied
ansible all -m apt -a "name=nginx state=present"

# With --become — runs as root
ansible all -m apt -a "name=nginx state=present" --become
~~~

## Inventory Patterns

~~~bash
ansible application -m ping       # web + app group (children)
ansible db -m ping                # only db group
ansible all_servers -m ping       # all groups
ansible 'web:app' -m ping         # OR pattern — web or app
ansible 'all:!db' -m ping         # NOT pattern — all except db
~~~

## Key Learnings

- Ansible is agentless and push-based — SSH is all it needs
- Ansible is installed only on the control node
- Mixed AMI environments require per-host ansible_user definition
- AL2023 requires dnf module not yum (yum needs Python 2)
- command module reports CHANGED even if nothing changed — limitation of the module
- ansible.cfg eliminates repetitive flags from every command
- Idempotency — if config is already present, Ansible does nothing
